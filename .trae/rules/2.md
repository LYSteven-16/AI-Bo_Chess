# AI 助手全局规则

> 适用于所有项目，优先级低于项目级规则

---

## 🗣️ 语言规范

- **始终使用中文回复**
- 思考过程、调用工具、分析代码、制定计划等**所有步骤**必须使用中文

---

## 📁 项目规则加载顺序

| 优先级 | 来源 | 范围 |
|--------|------|------|
| 1 | 全局规则（本文件） | 所有项目 |
| 2 | 项目规则（`.trae/rules/*.md` 或 `.claude/skills/`） | 当前项目 |

> ⚠️ 如果项目里有同名规则，**以项目规则为准**

---

## 📋 工作流程（必须遵守）

### 阶段 1：分析 → 穷举根因

- 读取相关源码，找到**根因**（从代码中找证据，**不要猜**）
- 根因必须穷举，每次穷举完成后要重新检查是否穷举完成
- 附上文件路径和行号

### 阶段 2：规划 → 输出后等待确认

```markdown
## 🎯 目标
（一句话说清楚要做什么）

## 🔍 根因
（从代码中找到的根本原因）
- 附文件路径和行号
- 每个可能的原因都要列出来

## 📝 实施步骤
1. 改文件 A：具体改什么 + 为什么
2. 改文件 B：具体改什么 + 为什么

## ✅ 验收标准
（可验证的标准）
```

### 阶段 3：执行

- 获得用户确认后再动手
- 按计划逐步实施

### 阶段 4：验证

- 按验收标准逐条检查
- 构建测试通过后再报告

---

## 📤 验证完成后的输出格式

```markdown
✅ 验收结果：
- [x] 标准1 ✅ 已通过
- [x] 标准2 ✅ 已通过
- [ ] 标准3 ❌ 未通过（原因：...）
```

---

## 🎯 技能使用参考

| 用户需求 | AI 动作 |
|----------|---------|
| "验证一下这个按钮" | 读取对应技能 → 构建 → 测试 → 报告 |
| "帮我 review PR" | 读取对应技能 → 分析 → 输出报告 |
| "有 bug，报错 XXX" | 读取对应技能 → 定位根因 → 给出方案 |
| "我要提交改动" | 读取对应技能 → 先展示 message → 确认后提交 |
| "10 个文件都要加日志" | 读取对应技能 → 拆解任务 → 并行执行 |
| "代码有点乱" | 读取对应技能 → 审查 → 指出问题 |

---

## ✏️ 编辑原则

1. **精准修改**：只改需要改的部分，不重写整个文件
2. **使用 edit 工具**：提供 `old_string` + `new_string` 配对
3. **先读后改**：修改前先 read 文件理解上下文
4. **遵循风格**：保持项目现有代码风格一致

## 🚫 约束

| 情况 | 做法 |
|------|------|
| 有疑问 | 先问再动手 |
| 破坏性操作 | 不执行 |
| 私人信息 | 不泄露 |
